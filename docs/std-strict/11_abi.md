# 11. 调用约定 (ABI)

调用约定（ABI，Application Binary Interface）规定了函数调用双方如何配合使用寄存器和栈。Std-Strict 的核心设计决定是：**把 ABI 从隐式约定变成显式语法**，让契约完整地写在函数签名里。

## 签名即完整契约

函数签名包含三部分，共同组成 ABI 契约：

```rust
pub fn process(r1, r2) -> r0 uses (r3, r4)
                ^^^^^^     ^^         ^^^^^^
                参数        返回       临时寄存器
```

这份契约告诉双方：

- **调用者必须做的**：把参数放进 `r1` 和 `r2` 再调用
- **被调用者承诺的**：返回值会放在 `r0` 里
- **调用者应当预期的**：调用后 `r0`、`r3`、`r4` 的内容都不可信
- **被调用者承诺的**：除了 `r0`、`r3`、`r4`，其他寄存器在调用前后保持不变

## 三类寄存器

在任何一次函数调用中，每个寄存器会处于下列三种角色之一：

### 参数 / 返回值寄存器

出现在函数签名的参数列表或 `->` 后。调用双方按照签名规定的顺序使用。

### 临时寄存器（Caller-saved，出现在 `uses` 中）

被调用者可以自由读写这些寄存器，调用者**不能**假定它们在调用后保持原值。如果调用者需要保留临时寄存器的值，必须在调用前自行保存：

```rust
// 假设要调用 process(r1, r2) -> r0 uses (r3, r4)
// 而我想在调用后继续使用 r3

r5 = r3                               // 先把 r3 保存到未被 use 的 r5
(r0) = process(r1 = ..., r2 = ...)    // 调用（r3 被破坏）
r3 = r5                                // 恢复 r3
```

### Callee-saved 寄存器（未出现在签名任何位置）

**没有出现**在签名任何位置的寄存器都属于 callee-saved。被调用者如果要使用它们，必须**自己保存和恢复**：

```rust
pub fn foo(r1) -> r0 uses r2 {
    // 这里我想用 r5，但没在签名中声明
    // 必须在使用前保存、返回前恢复
    
    r9 = stack.alloc(8)
    mem.i64[r9] = r5           // 保存原始的 r5

    r5 = r1 * 2
    r2 = r5 + 1
    r0 = r2

    r5 = mem.i64[r9]           // 恢复 r5
}
```

这样调用者就能放心地相信 `r5` 在调用前后保持不变。

## uses 子句是强制的

只要函数体用到了任何"参数和返回值之外的寄存器"，就**必须**在 `uses` 子句中声明。编译器会对函数体里的寄存器写入做静态检查：

| 寄存器属于         | 函数体内允许写吗？         |
|--------------------|----------------------------|
| 参数寄存器         | 允许                       |
| 返回值寄存器       | 必须写（返回前需要赋值）   |
| `uses` 列表中的    | 允许                       |
| 其他寄存器         | **禁止**（除非自行保存/恢复） |

违反规则的例子：

```rust
pub fn bad(r1) -> r0 {
    r5 = r1 + 1        // 编译错误：r5 未在签名中声明
    r0 = r5
}
```

修正方式之一——把 `r5` 加入 `uses`：

```rust
pub fn good(r1) -> r0 uses r5 {
    r5 = r1 + 1
    r0 = r5
}
```

修正方式之二——用 callee-saved 方式保存/恢复（当你想保持签名简洁时）。

## 没有自动推导

`uses` **不会**被编译器自动推导。这是刻意的设计：签名是调用双方共同看到的契约，让编译器悄悄补全会让 ABI 变成函数体的副作用，而 ABI 应该是明确声明的意图。

如果你希望编译器帮你列出 `uses`，可以通过 IDE / 工具链的 "quick fix" 功能——它会在编译错误后建议完整的 `uses` 列表让你粘贴。但源码本身必须显式写清楚。

## 多返回值的 ABI

多返回值在 ABI 层面和多个独立的"调用后活寄存器"没有本质区别——只是它们被承诺持有具体的值，而不是被破坏的垃圾：

```rust
fn divmod(r1, r2) -> (r0, r3) uses r4 {
    r0 = r1 / r2
    r3 = r1 % r2
    r4 = r0 * r2           // 临时用一下 r4，没问题
    // r4 的值不用写回签名声明的返回寄存器
}
```

从调用者视角：

- 调用后 `r0` = 商，`r3` = 余数
- 调用后 `r4` 的内容失效（因为在 `uses` 里）
- 其他寄存器在调用前后保持不变

## 栈传参

当参数数量超过可用寄存器时（或者出于 ABI 兼容性需要），可以把部分参数放在栈上。语法形式为 `stack[<offset>]`：

```rust
pub fn many_args(r1, r2, r3, r4, stack[0], stack[8]) -> r0 {
    // r1 ~ r4 通过寄存器传入
    // 第 5 个参数在 sp + 0
    // 第 6 个参数在 sp + 8
    r5 = mem.i64[stack[0]]
    r6 = mem.i64[stack[8]]
    // ...
}
```

调用者需要在调用前把值写入对应的栈槽。具体机制参见 [17. 栈管理](./17_stack.md)。

## 与 Native-Strict 的关系

Std-Strict 的 ABI 契约是**抽象的**——它说的是 `r1`、`r2`、`r0` 这些虚拟名字。在降级到 Native-Strict 时，这些虚拟寄存器会被映射到具体平台的物理寄存器：

| Std-Strict    | x86-64 (System V) | AArch64 | RISC-V |
|---------------|-------------------|---------|--------|
| `r0` (返回值) | `rax`             | `x0`    | `a0`   |
| `r1` (参数 1) | `rdi`             | `x0`    | `a0`   |
| `r2` (参数 2) | `rsi`             | `x1`    | `a1`   |
| `r3` (参数 3) | `rdx`             | `x2`    | `a2`   |
| ...           | ...               | ...     | ...    |

具体映射表在 Native-Strict 文档中给出。Std-Strict 代码**不需要关心这些映射**——它只需要遵守自己声明的虚拟 ABI。

## 为什么要这么设计

传统汇编里 ABI 是一种约定俗成的"文化"——写代码的人需要记住 System V AMD64 的参数顺序是 `rdi, rsi, rdx, rcx, r8, r9`，记住 `rbx` 是 callee-saved 而 `rcx` 是 caller-saved。一旦搞错，程序会以各种诡异的方式崩溃。

Std-Strict 把这些约定提升为**语言层面的语法**：

- 参数列表直接写在函数名后
- 返回值用 `->` 标明
- 临时寄存器用 `uses` 显式列出
- 调用处用命名参数 `fn(r1 = ...)` 让铺寄存器的动作可见

代价是每个函数签名多几个字，收益是你彻底不会再写出"忘记保存 callee-saved 寄存器"或者"调用前忘了设 r2"这类 bug——编译器会帮你抓住。
