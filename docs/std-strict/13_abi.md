# 13. 调用约定 (ABI)

调用约定（ABI，Application Binary Interface）规定了函数调用双方如何配合使用寄存器和栈。Std-Strict 的核心决定是：**把 ABI 从隐式约定变成显式语法**，让完整契约直接写在函数签名里。

## 签名即完整契约

函数签名包含四部分：

```rust
pub fn process(r1: i64, r2: *u8) -> (r0: i64) uses (r3, r4)
                ^^^^^^^^^^^^^^^^     ^^^^^^^^^^         ^^^^^^
                参数 + 类型           返回 + 类型         临时寄存器（不带类型）
```

这份契约告诉双方：

- **调用者必须做的**：把一个 `i64` 放入 `r1`，把一个 `*u8` 放入 `r2`，再调用
- **被调用者承诺的**：返回后 `r0` 里是 `i64`
- **调用者应当预期的**：调用后 `r0`、`r3`、`r4` 的内容都不可信
- **被调用者承诺的**：除了 `r0`、`r3`、`r4`，其他寄存器在调用前后保持不变

### 别名是可读层，契约仍是槽 + 类型

签名可以用 `@` 给参数 / 返回 / `uses` 的槽起别名（见 [05. 寄存器别名](./05_registers.md#寄存器别名)）：

```rust
pub fn process(input @ r1: i64, buf @ r2: *u8)
    -> (result @ r0: i64)
    uses (tmp @ r3, acc @ r4)
```

别名**只改变源码的可读性**——机械的 ABI 契约仍然是"槽 + 类型 + `uses`"。两个签名，只要**槽**（`r1`/`r2`/`r0`/`r3`/`r4`）、**类型**（`i64`/`*u8`/`i64`）、**`uses` 集合**完全相同，就是**同一个 ABI**，可以互相作为函数指针赋值，可以在 FFI 导出后被 C 端按同一套规则调用。调用者看到签名里的别名后必须用别名写命名参数（`process(input = ..., buf = ...)`），但底层寄存器分配、callee-saved 集合、参数传递规则一字不差。

## 三类寄存器

在任何一次函数调用中，每个寄存器处于下列三种角色之一：

### 参数 / 返回值寄存器

出现在函数签名的参数列表或 `->` 后。带类型。调用双方按签名规定的顺序和类型使用。

### 临时寄存器（Caller-saved，出现在 `uses` 中）

被调用者可以自由读写这些寄存器，调用者**不能**假定它们在调用后保持原值。**`uses` 列表里不写类型**——临时寄存器的类型随体内赋值自由浮动。

如果调用者需要保留临时寄存器的值，必须在调用前自行保存：

```rust
r5 = r3                                        // 先把 r3 保存到未被 use 的 r5
(r0) = process(r1 = ..., r2 = ...)             // 调用（r3 被破坏）
r3 = r5                                         // 恢复 r3
```

### Callee-saved 寄存器（未出现在签名任何位置）

**没有出现**在签名任何位置的寄存器都属于 callee-saved。被调用者如果要使用它们，必须**自己保存和恢复**：

```rust
pub fn foo(r1: i64) -> (r0: i64) uses r2 {
    // 想用 r5，但没在签名中声明
    r9 = stack::alloc<i64>(count = 1)
    mem.i64[r9] = r5                           // 保存
    r5 = r1 * 2
    r2 = r5 + 1
    r0 = r2
    r5 = mem.i64[r9]                           // 恢复
}
```

这样调用者就能放心地相信 `r5` 在调用前后保持不变。

## `uses` 子句是强制的

只要函数体用到了任何"参数和返回值之外的寄存器"，就**必须**在 `uses` 子句中声明。编译器会对函数体里的寄存器写入做静态检查：

| 寄存器属于         | 函数体内允许写吗？             |
|--------------------|--------------------------------|
| 参数寄存器         | 允许                           |
| 返回值寄存器       | 必须写（返回前需要赋值）       |
| `uses` 列表中的    | 允许                           |
| 其他寄存器         | **禁止**（除非自行保存/恢复）  |

违反规则的例子：

```rust
pub fn bad(r1: i64) -> (r0: i64) {
    r5 = r1 + 1        // 编译错：r5 未在签名中声明
    r0 = r5
}
```

修正方式之一——把 `r5` 加入 `uses`：

```rust
pub fn good(r1: i64) -> (r0: i64) uses r5 {
    r5 = r1 + 1
    r0 = r5
}
```

### 没有自动推导

`uses` **不会**被编译器自动推导。签名是调用双方共同看到的契约，让编译器悄悄补全会把 ABI 变成函数体的副作用——而 ABI 应该是明确声明的意图。

IDE / 工具链可以提供"quick fix"在编译错误后建议完整的 `uses`，但源码本身必须显式写清楚。

## 寄存器类型流（Dreg）

这是 Std-Strict 的类型系统核心：**每个程序点，每个活跃寄存器都有一个"当前类型"，由最近一次对它的赋值决定**。

五条规则覆盖了所有情况：

### 规则 1：入口

参数寄存器在函数入口的当前类型 = 签名声明的类型。

```rust
fn foo(r1: i64, f1: f32) -> (r0: i64) {
    // 此刻 r1 的当前类型 = i64, f1 的当前类型 = f32
}
```

### 规则 2：赋值即遮蔽

`rX = <expr>` 之后，`rX` 的当前类型 = `<expr>` 的类型——无需 `as`，自动更新（类比 Rust `let x = ...` 的重绑定）：

```rust
fn demo(r1: i64) -> (r0: i64) uses (r2, r3) {
    r2 = mem.u8[r1]                                // r2: u8
    r2 = conv::zero_extend_from_u8(r2)             // r2 遮蔽为 i64
    r3 = 1.5f64                                    // 整数寄存器存 f64？编译错
}
```

**整数寄存器不能承载浮点类型，浮点寄存器不能承载整数/指针类型**——寄存器类别（`r*` vs `f*`）仍然强约束。遮蔽只在同一类别内发生。

### 规则 3：操作按当前类型检查

算术、比较、内存下标、函数调用实参——所有读寄存器的操作都按它**此刻**的当前类型做类型检查。跨位宽（如 `i64 ↔ u8`）需要显式 `as` 或走 `std/convert`。

```rust
r1: i64
r2 = mem.u8[r1]                    // r2: u8
r3 = r1 + r2                       // 编译错：i64 + u8 位宽不同
r3 = r1 + (r2 as i64)              // OK
r3 = r1 + conv::zero_extend_from_u8(r2)     // OK（推荐）
```

字面量的处理详见 [06. 字面量](./06_literals.md)——简单说：无后缀字面量**跟随另一个操作数的类型**。

### 规则 4：合流一致

控制流合流点上，每个前驱分支对同一寄存器的当前类型必须一致，否则编译错：

```rust
fn bad(r1: i64) -> (r0: i64) uses r2 {
    if (r1 > 0) goto(@hot)
    r2 = -r1                      // r2: i64
    goto(@join)
@hot:
    r2 = 42u8                     // r2: u8
@join:
    r0 = r2
    // 编译错：r2 在 @hot 路径是 u8，在另一路径是 i64，合流点不一致
}
```

**没有赋值的路径**依然"带着进入分支前的类型"到合流点——所以哪怕只有一条分支改了类型，只要另一条没改、与改过的那条不一致，就会报错。

### 规则 5：出口

`return` 时（含函数末尾的隐式返回），所有返回寄存器的当前类型必须等于签名声明。不匹配编译错：

```rust
fn bad2(r1: i64) -> (r0: i64) uses r5 {
    r0 = 42u8             // 编译错（在函数末尾）：r0 当前是 u8，签名要 i64
}
```

## 调用点检查

签名参数类型驱动实参类型检查：

```rust
fn takes_i64(r1: i64) -> (r0: i64) { ... }

r5 = mem.u8[r1]                                   // r5 当前类型 = u8
takes_i64(r1 = r5)                                // 编译错
takes_i64(r1 = r5 as i64)                         // OK
takes_i64(r1 = conv::zero_extend_from_u8(r5))     // OK（推荐）
```

返回值的类型也由签名决定——多返回解构后，每个寄存器的当前类型 = 对应位置的签名类型。

## `#[builtin]` 的有限泛型

编译器内置函数可以声明 `<T>` 类型参数：

```rust
#[builtin] pub fn select<T>(cond: bool, if_true: T, if_false: T) -> T
```

- 用户定义的 `fn` **不能**写泛型
- 编译器根据每个调用点的实参推导 `T`，为每种实例生成一份实现
- **同一 `T` 绑定的所有实参必须严格同类型**——`select(r2, 1i32, 2i64)` 编译错（两个分支类型不一致）

详见 [24. 内置库总览](./24_builtins_overview.md)。

## 多返回值的 ABI

多返回值在 ABI 层面就是多个独立的"调用后活寄存器"——只是它们持有具体的值而非垃圾：

```rust
fn divmod(r1: i64, r2: i64) -> (r0: i64, r3: i64) uses r4 {
    r0 = r1 / r2
    r3 = r1 % r2
    r4 = r0 * r2           // 临时用一下 r4，没问题
}
```

从调用者视角：

- 调用后 `r0` = 商（i64），`r3` = 余数（i64）
- 调用后 `r4` 的内容失效（因为在 `uses` 里）
- 其他寄存器在调用前后保持不变

## 栈传参

当参数数量超过可用寄存器时（或者出于 ABI 兼容性需要），可以把部分参数放在栈上。语法形式 `stack[<offset>]`：

```rust
pub fn many_args(r1: i64, r2: i64, r3: i64, r4: i64, stack[0]: i64, stack[8]: *u8) -> (r0: i64) {
    r5 = mem.i64[stack[0]]
    r6 = mem.i64[stack[8]]
}
```

调用者在调用前把值写入对应的栈槽。具体机制参见 [32. std/stack](./32_std_stack.md)。

## 与 Native-Strict 的关系

Std-Strict 的 ABI 契约是**抽象的**——它说的是 `r1`、`r2`、`r0` 这些虚拟名字。降级到 Native-Strict 时，这些虚拟寄存器会被映射到具体平台的物理寄存器：

| Std-Strict    | x86-64 (System V) | AArch64 | RISC-V |
|---------------|-------------------|---------|--------|
| `r0` (返回值) | `rax`             | `x0`    | `a0`   |
| `r1` (参数 1) | `rdi`             | `x0`    | `a0`   |
| `r2` (参数 2) | `rsi`             | `x1`    | `a1`   |
| `r3` (参数 3) | `rdx`             | `x2`    | `a2`   |
| `f0` (返回值) | `xmm0`            | `v0`    | `fa0`  |
| `f1` (参数 1) | `xmm0`            | `v0`    | `fa0`  |

具体映射表在 Native-Strict 文档中给出。Std-Strict 代码**不需要关心这些映射**——它只遵守自己声明的虚拟 ABI。

## 为什么要这么设计

传统汇编里 ABI 是一种约定俗成的"文化"——写代码的人需要记住 System V AMD64 的参数顺序是 `rdi, rsi, rdx, rcx, r8, r9`，记住 `rbx` 是 callee-saved 而 `rcx` 是 caller-saved。一旦搞错，程序以各种诡异方式崩溃。

Std-Strict 把这些约定提升为**语言层面的语法**：

- 参数列表直接写在函数名后，带类型
- 返回值用 `->` 标明，带类型
- 临时寄存器用 `uses` 显式列出
- 调用处用命名参数让铺寄存器的动作可见
- 每个程序点的当前类型由编译器跟踪，类型流错误静态可捕获

代价是每个函数签名多几个字，收益是"忘记保存 callee-saved"或"调用前忘了设 r2"或"把 u8 当 i64 传过去"这类 bug 都编译不过去。
