# 05. 虚拟寄存器

Std-Strict 不存在传统意义上的"变量"。所有运行时状态都承载在**寄存器**里。本章讲清楚寄存器的使用规则。

  已覆盖：`rN/fN` 引用、签名别名（`name @ rN: T`）、文件级/函数体级 `alias`、多目标赋值左值元组、参数化标签相关寄存器位点。  
  未覆盖：全部寄存器相关语法细节与 native-strict 专属约束。

  未覆盖：完整 Dreg 类型流与复杂控制流合流点校验。
- 编译（Windows x86_64）：`[~]` 部分完成（最小寄存器映射已接通，完整分配器仍在后续阶段）

## 两个寄存器文件

Std-Strict 提供两组独立的虚拟寄存器：

| 名称范围       | 类别    | 容量    | 用途                         |
|----------------|---------|---------|------------------------------|
| `r0` – `r999`  | 整数/地址 | 64 位  | 整数运算、地址、布尔、指针   |
| `f0` – `f999`  | 浮点    | 64 位   | 承载 `f32` 或 `f64` 值       |

```rust
r0
r42
r999
f0
f7
f128
```

这只是**命名约定**，不代表目标机器上真的有 1000 个物理寄存器。编译器后端在代码生成阶段会把这些虚拟名字映射到目标架构的物理寄存器，用不完的部分会溢出（spill）到栈上。

整数寄存器和浮点寄存器是**不同的命名空间**——不能混用，也不能互相赋值：

```rust
r1 = f1                                // 编译错：跨类别赋值
r1 = conv::bitcast_f64_to_i64(f1)      // OK：显式位重解释
```

## 寄存器槽 vs 当前类型

关于"类型"这件事，Std-Strict 有两层视角：

### 物理视角：槽是无类型的

寄存器物理上就是 64 位的存储槽，里面的比特可以代表任何东西——整数、浮点的位表示、地址、布尔。

### 编译器视角：每个程序点都有"当前类型"

在任何一个程序点，编译器都会给每个活跃的寄存器记账一个**当前类型**。这个当前类型由**最近一次对它的赋值**决定：

```rust
r1 = 42i64               // r1 的当前类型 = i64
r1 = mem.u8[r5]          // r1 的当前类型变成 u8（隐式遮蔽）
r1 = &G_TABLE            // r1 的当前类型变成 *u8
```

**每次赋值都会重新定义寄存器的当前类型**——这和 Rust 里 `let x = 1; let x = "hi";` 的遮蔽（shadowing）一样。不需要 `as` 转换，因为寄存器是存储槽不是变量。

这种"逐次赋值、逐次换型"的语义意味着：

- 读寄存器时，按照它**最新被赋的类型**做类型检查
- 算术/比较/内存下标等操作要求参与者的当前类型互相匹配（或满足字面量的上下文推导）
- 跨位宽仍要显式 `as` 或走 `std/convert`；**当前类型的改变来自于"被赋了新类型的值"，不是凭空换**

完整的类型流规则见 [13. 调用约定 ABI](./13_abi.md#寄存器类型流dreg)。

## 赋值

最基础的寄存器操作：

```rust
r1 = 42i64                 // 立即数（带后缀）
r1 = 42                    // 立即数（类型由上下文推导）
r2 = r1                    // 寄存器到寄存器
r3 = r1 + r2               // 表达式求值
r4 = &G_TABLE              // 取地址
r5 = mem.i32[r4]           // 内存读取
f1 = 3.14f64               // 浮点立即数
f2 = mem.f64[r4]           // 浮点内存读取
```

赋值目标（左值）必须是寄存器或内存位置，不能是立即数或常量。

## 调用约定对寄存器的约束

虽然寄存器在语法上都是平等的，但在跨函数调用时，它们的语义受 [调用约定](./13_abi.md) 约束：

- **参数寄存器**：调用者负责在调用前设置
- **返回值寄存器**：被调用者负责在返回前设置
- **`uses` 列表**：被调用者可以自由使用，调用者不应假定它们在调用后还保留原值
- **未声明的寄存器**：函数体内禁止写入（除非显式保存/恢复）

这些规则由每个函数的签名显式声明，而不是隐藏在语言规范里。详见 [12. 函数](./12_functions.md) 和 [13. 调用约定](./13_abi.md)。

## 特殊寄存器

Std-Strict 故意**不暴露**以下寄存器：

- 栈指针 `sp`
- 帧指针 `bp`
- 指令指针 `pc`
- 标志位寄存器 `flags`

栈的使用通过 [`std/stack`](./32_std_stack.md) 完成，控制流通过 [`goto` 和条件跳转](./11_control_flow.md) 完成。这样做让同一份 `.ap` 代码可以无差别地编译到 x86、ARM、RISC-V 以及 WebAssembly（后者根本没有硬件栈指针）。

## 寄存器别名

`r1` / `f3` 这种按编号命名的寄存器写起来紧凑，但函数一长就变成密码——你得靠注释或记忆才能知道"r5 是什么"。Std-Strict 允许给寄存器槽起名字：**别名**。别名只是词法层的重命名，不改变 Dreg 的类型流、ABI 契约或运行时语义。

### 三种声明位置

别名的声明都通过 `@` 绑定——`name @ slot` 把标识符 `name` 绑定到寄存器槽 `slot`。

#### 1. 签名级（参数 / 返回 / `uses`）

直接写在函数签名里。这是最常用的形式：

```rust
fn add(count @ r1: i64, step @ r2: i64)
    -> (sum @ r0: i64)
    uses (tmp @ r5, acc @ r6)
{
    sum = count + step
    tmp = step * 2i64
    acc = sum - tmp
    sum = acc
}
```

`name @ slot: type` 的三要素：

- `name`——在函数体内用的名字
- `@ slot`——绑定到哪个 `r<n>` / `f<n>`
- `: type`（参数 / 返回）或无（`uses`）——沿用原本的类型规则

#### 2. 函数体级

必须出现在函数体**最开头**——所有 `alias` 声明必须在任何一条非别名语句之前。之后再写 `alias` 是编译错：

```rust
pub fn foo(r1: i64, r2: i64) -> (r0: i64) uses (r3, r4) {
    alias count @ r1
    alias step @ r2
    alias sum @ r0
    alias tmp @ r3

    sum = count + step
    tmp = count * 2i64
}
```

这种形式常见于签名保持"裸 `r*`"风格（和其它函数签名对齐）但函数体内想要可读变量名的情况。

#### 3. 文件级

放在文件**开头**，在任何 `fn` / `val` / `const` / `struct` / `type` / `extern fn` 之前——只有 `import` 和注释可以在 `alias` 声明之前：

```rust
import "std/io" as io

alias ERRNO @ r50
alias GLOBAL_FLAGS @ r51

pub fn read(...) { ... }
pub fn write(...) { ... }
```

文件级别名对**本文件内**所有函数生效，用于建立"这个模块内 r50 永远叫 ERRNO"这类约定。文件级别名**不导出**、不产生链接器符号——它纯粹是源码层面的命名。

### 严格替换

一旦一个槽有了活跃的别名，**该作用域内写原始 `r<n>` / `f<n>` 是编译错**：

```rust
pub fn foo(count @ r1: i64) -> (r0: i64) {
    r0 = r1 + 1i64              // 编译错：r1 在本函数里叫 count
    r0 = count + 1i64           // OK
}
```

这样做让"同一个函数里一会儿 `count` 一会儿 `r1`"的混乱风格不可能发生。

### 作用域覆盖

三种作用域由外到内：**文件级 → 签名级 → 函数体级**。内层 `@` 绑定可以**重新绑定**外层别名涉及的槽，重新绑定期间外层名字不可见：

```rust
alias ERRNO @ r50                          // 文件级

// 签名重新绑定 r50 → 本函数里 r50 叫 retval；ERRNO 在本函数不可见
pub fn syscall_raw(num @ r1: u64) -> (retval @ r50: i64) {
    retval = 0i64
}

// 不重新绑定 → 沿用文件级 ERRNO
pub fn set_errno(code @ r1: i64) uses (ERRNO) {
    ERRNO = code
}
```

函数体 `alias` 覆盖签名里的别名名的话，按"先签名、后体级"的顺序声明：

```rust
fn foo(count @ r1: i64) -> (r0: i64) {
    alias n @ r1                // 合法：把 count 重命名为 n
    // 此后只能用 n，count 不再可见
    r0 = n * 2i64
}
```

> 不过多数时候没必要——直接在签名里写 `n @ r1` 即可。

### 命名约束

别名名字遵循普通标识符规则，并且**不得**：

- 撞关键字（`fn` / `if` / `goto` / `alias` / ...）
- 撞保留名字（`r0`–`r999`、`f0`–`f999`、`mem`、`stack`、`true`、`false` 等）
- 撞同作用域内已存在的别名
- 撞同作用域内可见的 `const` / `val` / `var` / `fn` / `struct` / `type` / `import` 别名
- 同一个槽在同一作用域有两个活跃别名（一个槽、一个名字）

按约定，寄存器别名用小写蛇形（`count` / `tmp_buffer`）以便和常量 `MAX_SIZE` / 类型 `Point` 区分。

### 对其他构造的影响

别名是词法层的替换，对下列构造**完全透明**：

```rust
alias buf @ r1
alias i   @ r2

mem.u8[buf + i]       = 0u8              // mem.<T>[alias] 照常
r0                    = mem.i64[buf]     // 混用：OK，r0 未被别名

(count, rem) = x / y                     // 多目标赋值用别名
fn_ptr(buf = ..., i = ...)               // 命名参数用别名
if (count > 0) goto(@loop)               // 条件、跳转、标签都 OK
```

别名**不能**用作标签名（`@loop` 的 `loop` 和 `alias loop @ r5` 的 `loop` 处于不同命名空间——但为了可读性，别名和标签名最好仍不重名）。

### 对调用点的影响

签名里的别名**成为参数的规范名**——调用方必须用别名：

```rust
fn add(count @ r1: i64, step @ r2: i64) -> (sum @ r0: i64) { ... }

// 调用
r0 = add(count = 5i64, step = 3i64)      // OK
r0 = add(r1 = 5i64,    r2 = 3i64)        // 编译错：签名里 r1/r2 叫 count/step
```

这让签名成为单一事实来源——想换参数寄存器只改签名，调用端不动。

### 对函数指针类型兼容性

**类型兼容性忽略别名名字**——只看槽 + 类型 + `uses`：

```rust
type BinOp = fn(r1: i64, r2: i64) -> (r0: i64)

fn add(count @ r1: i64, step @ r2: i64) -> (sum @ r0: i64) {
    sum = count + step
}

pub fn pick() -> (r0: BinOp) {
    r0 = &add                            // OK：槽和类型匹配，别名不参与判定
}
```

### 对 `extern fn` / `#[builtin]` / 宏

- `extern fn` 签名可以用 `@` 给槽命名——更可读，但对底层 C ABI 无影响（C 函数的"形参名"本来就只是元数据）
- `#[builtin]` 签名里的形参名（比如 `divmod` 的 `dividend` / `divisor`）**等价于**用户函数的别名名——调用点的命名参数规则一致
- 宏的 `reg` 片段匹配到别名时，解析为底层槽，展开后按槽参与类型流

### 对 `#[naked]`

`#[naked]` 函数里**禁止**使用别名——别名声明（签名 `@` 或体级 `alias`）在 `#[naked]` 函数里是编译错。`#[naked]` 关掉了 Dreg 的内部检查，再加别名没意义，而且会让"到底哪个寄存器被 clobber"这件事模糊。

### 对照示例

同一个函数，纯 `r*` 版：

```rust
pub fn strlen(r1: *u8) -> (r0: u64) uses (r2) {
    r0 = 0u64
@loop:
    r2 = mem.u8[r1 + r0]
    if (r2 == 0u8) goto(@done)
    r0 = r0 + 1u64
    goto(@loop)
@done:
}
```

别名版：

```rust
pub fn strlen(str @ r1: *u8) -> (len @ r0: u64) uses (byte @ r2) {
    len = 0u64
@loop:
    byte = mem.u8[str + len]
    if (byte == 0u8) goto(@done)
    len = len + 1u64
    goto(@loop)
@done:
}
```

生成的机器码完全一致，但读者一眼看懂"`len` 是长度、`byte` 是字节、`str` 是字符串指针"。
