# 15. 内联函数

`inline` 告诉编译器：**在调用处直接展开这个函数的函数体**，不生成真正的 `call` 指令。

## 语法

两种等价写法：

```rust
inline fn add(r1: i64, r2: i64) -> (r0: i64) {
    r0 = r1 + r2
}

#[inline]
fn add(r1: i64, r2: i64) -> (r0: i64) {
    r0 = r1 + r2
}
```

可以与 `pub` / `export` 组合：

```rust
inline pub fn fast_mul(r1: i64, r2: i64) -> (r0: i64) {
    r0 = r1 * r2
}
```

## 行为

`inline` 函数**没有独立的机器码实体**——编译器不会为它生成入口地址，调用点也不会出现 `call` 指令。函数体直接嵌入到每一个调用处，类似于 C 的 `static inline`。

对下面这段代码：

```rust
inline fn double(r1: i64) -> (r0: i64) {
    r0 = r1 * 2
}

pub fn main() -> (r0: i64) {
    double(r1 = 10)
}
```

经过内联展开，相当于：

```rust
pub fn main() -> (r0: i64) {
    r1 = 10i64
    r0 = r1 * 2       // double 的函数体
}
```

## ABI 与类型流

**即使是 `inline` 函数，签名依然是契约**——调用方写的还是 ABI 风格的代码（命名参数），类型检查也按签名做。内联只是省掉了跳转，**不放松**寄存器使用规则和 Dreg 类型流检查。

## 限制

### 不能取地址

```rust
inline fn foo() -> (r0: i64) { r0 = 0 }

r1 = &foo        // 编译错：inline 函数没有实体地址
```

没有实体就没有地址——和 C 里对 `inline` 取地址一样尴尬。

### 不能递归

```rust
inline fn bad(r1: i64) -> (r0: i64) {
    bad(r1 = r1)        // 编译错：inline 函数不能直接或间接递归
}
```

展开递归函数会导致无限展开。

### 跨模块的 inline

`inline pub` 函数被另一个模块 `import` 时，**函数体必须对导入方可见**——编译器会把 inline 函数的源码级 AST 随符号一起缓存到模块接口里。这和 C++ 的 header-only inline 行为类似。

### 不能 `#[naked]`

`inline` + `#[naked]` 互斥。

## 什么时候用 inline

- 极小的函数（`r0 = r1 + r2` 这种），函数调用开销比函数本身还大
- 性能热点中的辅助函数
- 编译期常量参数参与运算（内联后可能被进一步常量折叠）

什么时候**不**用：

- 函数体很大，内联显著增大代码体积
- 需要函数指针（`&fn`）的场景
- 函数频繁递归

## inline vs 宏

`inline` 函数是"零开销函数"——类型检查、作用域、调试信息都按函数走。

[宏](./23_macros.md) 是"源码展开"——不是函数，不走 ABI，可以接受表达式 / 寄存器 / 字面量等非值的参数。

判断准则：能用函数表达就用 `inline fn`；需要在调用点拿到字面量的编译期值、需要 lazy 求值、需要一条语句块作为参数——用宏。
