# 07. 内存访问

内存在 Aperio 的视角里就是一片字节海，访问它需要三个信息：**地址**、**读写宽度**、**符号性**。本章介绍访问内存的语法。

## 基本语法

内存访问的通用形式：

```
mem.<type>[<address-expression>]
```

- `<type>` 是 [基础数据类型](./04_types.md) 之一：`i8` / `i16` / `i32` / `i64` / `u8` / `u16` / `u32` / `u64` / `f32` / `f64`
- `<address-expression>` 是一个产生地址的表达式

## 读取内存

`mem.<type>[addr]` 出现在赋值**右侧**时表示读取：

```text
r2 = mem.i32[r1]        // 从 r1 读 4 字节有符号整数
r3 = mem.u8[r1]         // 从 r1 读 1 字节无符号整数
r4 = mem.f64[r1]        // 从 r1 读 8 字节双精度浮点
```

读窄于 64 位的类型时，高位按符号性自动扩展（`i*` 用符号扩展，`u*` 用零扩展）。这一点在 [04. 数据类型](./04_types.md) 里讲过。

## 写入内存

`mem.<type>[addr]` 出现在赋值**左侧**时表示写入：

```text
mem.i32[r1] = r2        // 把 r2 的低 32 位写入 r1 指向的地址
mem.u8[r1] = r2         // 只写入 r2 的最低字节
mem.f64[r1] = r3        // 把 r3 的 8 字节写入
```

写入时高位会被直接**截断**，没有任何扩展或检查。

## 地址表达式

方括号里的地址表达式支持以下几种形式：

### 单个寄存器

```text
r2 = mem.i32[r1]
```

### 寄存器加常量偏移

```text
r2 = mem.i32[r1 + 8]
r2 = mem.i32[r1 - 4]
```

这对应 x86 的 `[rax + 8]` 寻址，非常适合访问结构体字段或数组。

### 寄存器加寄存器

```text
r2 = mem.i32[r1 + r3]
```

对应 x86 的 `[rax + rbx]`。

### 寄存器加缩放寄存器（数组索引）

```text
r2 = mem.i32[r1 + r3 * 4]
```

对应 x86 的 `[rax + rbx*4]`，用于按元素宽度索引数组。缩放因子只能是 `1`、`2`、`4` 或 `8`（这是 x86 的硬件限制，ARM 后端会转换为等价的左移 + 加法）。

### 完整形式

```text
r2 = mem.i32[r1 + r3 * 4 + 16]
```

对应 x86 的 `[rax + rbx*4 + 16]`。这是 x86 支持的最复杂寻址形式。

## 地址空间

默认的 `mem.<type>[...]` 等价于 `mem.global.<type>[...]`，表示全局可寻址的内存（进程的线性地址空间）。为了支持 CUDA 等非统一内存模型的目标，Aperio 预留了以下地址空间前缀：

| 前缀         | 含义                       |
|--------------|----------------------------|
| `mem.global` | 全局内存（默认）           |
| `mem.shared` | CUDA 共享内存（block 内）  |
| `mem.local`  | CUDA 线程局部内存          |
| `mem.const`  | CUDA / Wasm 常量内存       |

```text
r2 = mem.shared.i32[r1]
r3 = mem.const.f32[r1 + 16]
```

在编译目标为 CPU（x86 / ARM）时，除了 `global`，其他地址空间会被拒绝。

## 对齐

Aperio **不**对内存访问的对齐做任何强制要求，未对齐访问的行为由目标架构决定：

- x86 允许未对齐访问，但有性能代价
- ARM / RISC-V 多数情况下会触发异常
- Wasm 规范允许未对齐，但可能更慢

如果你需要保证对齐，应该在数据段定义时使用合适的类型和数组元素宽度，详见 [14. 数据段](./14_data_segments.md)。
