# 38. 未定义行为（附录）

Std-Strict 是一门**明确暴露硬件细节**的语言。和 C 一样，某些操作的后果在语言层面**不定义**——编译器可以假设它们"不会发生"来做优化。一旦真的发生了，你得到的可能是错误结果、崩溃、或者看起来正常但其实某行代码被"优化掉"了。

本附录列出所有 UB 条目。写 `.ap` 代码时要有意识地避开它们。

## 1. 内存访问类

### 1.1 越界访问

读写超出对象（`val` / `var` / `stack::alloc` / 堆对象 / 数组）边界的内存。

```rust
val ARR: i32[4]
r1 = &ARR                              // *i32
r2 = ptr::offset<i32>(r1, 10i64)       // OK：算地址不 UB
mem.i32[r2] = 1i32                     // UB：写到 ARR 外
```

### 1.2 悬空指针

访问已经释放的栈帧、已 `free` 的堆对象、或通过 `ptr::from_addr` 伪造的任意地址。

```rust
fn dangling() -> (r0: *i64) {
    r0 = stack::alloc<i64>(1u64)
}

pub fn use_it() uses r1 {
    r1 = dangling()
    mem.i64[r1] = 1i64                 // UB
}
```

### 1.3 对齐违规

用 `mem.<T>[p]` 读写的地址不满足 `T` 的自然对齐。

```rust
r1 = ptr::from_addr<i32>(0x1003u64)   // *i32 声称指向非 4 对齐地址
r2 = mem.i32[r1]                       // UB
```

例外：`mem.<T>.unaligned[p]` **明确允许**未对齐访问——不是 UB，见 [08. 内存访问](./08_memory.md)。

原子操作（`std/atomic`）的地址**强制**自然对齐，未对齐是编译错而非 UB（但若你用 `ptr::from_addr` 硬伪造一个未对齐地址再传进去，运行时是 UB）。

### 1.4 类型混淆

同一块内存同时被当作不同类型（不兼容的）访问，而没走 `bitcast_*`。

```rust
val F: f64 = 3.14
r1 = &F                                // *f64
r2 = ptr::cast<f64, i64>(r1)           // *i64
r3 = mem.i64[r2]                       // UB：strict aliasing 违规
```

安全的做法：用 `conv::bitcast_f64_to_i64` 把值过一遍转换，而不是通过 `mem.<i64>[...]` 读一个 `f64` 地址。

### 1.5 数据竞争

多个线程同时访问同一内存位置，至少一个是写，且**没有**用 `std/atomic` 或同步原语。

```rust
// 线程 A                  // 线程 B
mem.i64[r1] = 1i64         r2 = mem.i64[r1]         // UB
```

用 `atomic::load_i64_relaxed(r1)` / `atomic::store_i64_relaxed(r1, ...)` 规避。

## 2. 算术类

### 2.1 有符号整数溢出

`i8`/`i16`/`i32`/`i64` 的算术溢出（`+`/`-`/`*`/`<<`）。**Aperio 规定有符号溢出是 UB**，和 C 一致。

```rust
r1: i32 = 2147483647i32
r1 = r1 + 1i32                         // UB（编译器可能假设结果 ≥ r1）
```

需要回绕语义？用无符号版本：

```rust
r1: u32 = 2147483647u32
r1 = r1 + 1u32                         // OK：= 2147483648，确定性回绕
```

（字面量在表达式里由上下文推导为 `u32`，见 [06. 字面量](./06_literals.md)。）

### 2.2 无符号溢出

无符号算术**不是 UB**——按 2^N 回绕，行为完全确定。

### 2.3 除以零

`r1 / 0` 或 `r1 % 0`，无论有无符号都是 UB。

```rust
r1 = r2 / r3                           // 若 r3 = 0，UB
```

硬件行为（x86 的 `#DE` 异常、ARM 的零结果等）可能不一致；语言层面不做保证。

### 2.4 `INT_MIN / -1`

有符号整数的最小值除以 `-1`：

```rust
r1: i32 = 0x80000000i32               // INT32_MIN
r2 = r1 / -1i32                        // UB：数学结果 2^31 超出 i32
```

### 2.5 移位量过大或负

对 `N` 位整数做 `<<` / `>>`，移位量不在 `[0, N)` 范围内是 UB。

```rust
r1: i32 = 1i32
r2 = r1 << 32i32                       // UB（i32 的合法移位量是 0-31）
r3 = r1 << -1i32                       // UB
```

## 3. 浮点类（非 UB，但需注意）

浮点的"异常"（除零、溢出、NaN 传播）**不是 UB**——它们产生 IEEE 754 定义的 `±inf`/`NaN`：

```rust
f1 = 1.0 / 0.0                          // = +inf，不是 UB
f2 = 0.0 / 0.0                          // = NaN，不是 UB
f3 = f_max * 2.0                        // = +inf（不溢出为 UB）
```

`conv::f64_to_i32_trunc` 等浮点转整数：若值超出目标整数范围或是 NaN，结果是**平台定义**（x86 返回 `INT_MIN`，有些目标饱和），**不是 UB**。需要精确定义行为的场景自己先检查。

## 4. 控制流与函数类

### 4.1 调用 `#[noreturn]` 后返回

声明了 `#[noreturn]` 的函数实际走到了 `return`（或走到了函数体结束）是 UB：

```rust
#[noreturn]
pub fn shouldnt_return() {
    // 编译器相信这里不会返回
}
```

`panic!` / `os::exit` / `os::abort` / `debug::unreachable` 都是 `#[noreturn]` 的实现，违反约定的后果在它们自己的实现里。

### 4.2 到达 `unreachable`

`debug::unreachable()` 在逻辑上永远不应被执行。真的跑到了是 UB——编译器可能已经基于"不会跑到这里"把周围的代码折叠了。

### 4.3 函数指针不兼容

把一个函数通过不兼容的 `fn` 类型调用：

```rust
type F = fn(r1: i64) -> (r0: i64)
pub fn takes_u32(r1: u32) -> (r0: u32) { r0 = r1 }

r1: u64 = ptr::to_addr(&takes_u32)
r2: F = ptr::from_addr<F>(r1) as F    // 强转
r0 = r2(r1 = 42i64)                   // UB：实际签名 (u32) -> u32，调用约定不匹配
```

### 4.4 非 `#[naked]` 函数的 `uses` 声明不完整

用户函数实际修改了 `r5`，但 `uses` 里没列 `r5`：这是**编译错**，不是 UB。Dreg 会静态检查。所以这里实际上 UB 触发不到——编译器替你把关。

### 4.5 `#[naked]` 函数不守 ABI

`#[naked]` 关掉了 Dreg 的内部检查。由于编译器不再替你看守，任何违反 ABI 契约的行为（改了 callee-saved 寄存器、返回值放错位置）都是 UB。

## 5. 指针转换类

### 5.1 `*T` 和 `*U` 大小/对齐不兼容

```rust
r1: *u8 = &BUF                         // BUF: u8[...]
r2: *i64 = ptr::cast<u8, i64>(r1)      // 合法的类型重解释
r3 = mem.i64[r2]                       // 若 r1 的地址不满足 i64 对齐：UB
```

### 5.2 从 `u64` 造指针

```rust
r1: *i64 = ptr::from_addr<i64>(0x12345678u64)
mem.i64[r1]                            // UB：0x12345678 几乎肯定不是一个合法对象的地址
```

这类操作是故意允许的（MMIO、内核、原始 JIT 代码区），但语义归你负责。

## 6. 数据段与链接

### 6.1 在 `.rodata` 上写

```rust
const S: u8[] = c"hi"
r1 = &S
mem.u8[r1] = 0u8                       // UB：写只读段，硬件 SIGSEGV
```

### 6.2 `export` 符号名冲突

两个编译单元都 `export pub fn foo`：链接错。不是 UB，但需注意。

### 6.3 `extern fn` 签名与实际不符

`.ap` 里 `extern fn c_func(r1: i32) -> (r0: i32)`，但 C 里实际是 `double c_func(double, double)`：UB。FFI 签名由程序员负责匹配。

## 7. 运行期限制

某些操作在 `debug` 模式下会被编译器**插入运行时检查**并 panic，在 `release` 模式下走 UB 路径：

| 操作 | debug 行为 | release 行为 |
|------|-----------|--------------|
| 有符号溢出 | panic | UB |
| 除零 | panic | UB |
| 大移位量 | panic | UB |
| 越界 `mem.<T>[p]` 访问（若可静态推导范围） | panic | UB |

所以 debug 下跑过 ≠ release 下安全。关键路径应该配合 `debug::assert!` 做显式检查。

## 8. 编译期的 UB 防线

下列错误 Aperio 会在**编译期**拒绝，不会放进运行时 UB：

- Dreg 类型检查：寄存器类型不匹配、分支合并处类型冲突、`uses` 漏列
- 参数 / 返回值位置类型不匹配
- 常量表达式溢出
- 原子操作指针类型不对、对齐不对（编译期可推导时）
- `#[noreturn]` 函数里出现带值的 `return`
- 未定义模块、未定义符号、循环 import

这些早期检查是"非 UB"的主要来源——**愿意让编译器喊"不对"就不会掉进 UB 坑**。
